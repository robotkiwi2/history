# SPEC — 데이터 구조 / 점수 체계 / 동작 규칙

> 한국사 타임라인 게임의 현재 구현 기준 기술 사양. 변경 시 이 문서도 업데이트.
> 시간순 의사결정 로그는 [기획.md](기획.md) 참조.

---

## 1. 화면 구조

### 1.1 현재 구현
```
view-login        ← 시작 화면 (Google/카카오/네이버 소셜 버튼, OAuth 미구현)
  ↓ 어느 버튼이든 클릭
view-records      ← 메인 메뉴 + 내 기록 표시
  ├─ 우상단 ⚙ → view-settings (로그아웃, 기록 초기화)
  ├─ [연도] 버튼  → view-era → view-year (=Level 1, 키워드 연도순)
  └─ [사건] 버튼  → view-event (드래그 정렬, 별도 게임)
```

**뒤로가기 계층:**
- `view-year` ← `view-era` ← `view-records` ← `view-login`
- `view-event` ← `view-records`
- `view-settings` ← `view-records`

### 1.2 계획 (Level 1/2 분리, 디테일 매칭 모드 추가)
```
view-records
  ├─ [Level 1: 키워드 연도순] → view-era → view-year         (현재 연도 모드)
  ├─ [Level 2: 디테일 매칭]   → view-era(L2) → view-detail   (신규 — §8 참조)
  └─ [사건 정렬]              → view-event                    (현재 사건 모드)
```
- 메뉴 진입에서 명시적 Level 분기 (옵션 1A)
- Level 1과 Level 2는 같은 타임라인 데이터(키워드 카드 누적)를 공유 — Level 1에서 푼 키워드가 Level 2의 매칭 대상으로 등장

## 2. 카드 데이터 구조

### 2.1 `INITIAL_BG` — 연도 모드 기준 카드 (data.js)
```js
{ title: '500년', startYear: 500, endYear: 500 }
```
- 9장 고정: 기원전 2000 / 1500 / 1000 / 500 / 0 / 500 / 1000 / 1500 / 2000
- 모두 점(start=end), description·era·difficulty 없음
- 타임라인 뼈대 역할, 게임 시작 시 항상 같은 위치

### 2.2 `YEAR_QUESTIONS` — 연도 모드 문제 카드 (data.js)
```js
{
  title: '고려 건국',
  era: '고려시대',
  difficulty: 1,
  description: '왕건이 후삼국을 통일하고 세운 왕조의 시작',
  startYear: 918,
  endYear: 918,
}
```
- 약 200개
- `era`: 카드 데이터엔 raw 시대명(`'조선시대'`, `'근대'` 등) 저장. 표시·필터는 `cardEra(c)` 헬퍼가 startYear 기반으로 세분 매핑.
- **세분 시대 (10개)** — 한능검 출제 비중 반영:
  | 표시 시대 | 매핑 규칙 |
  |---|---|
  | 선사·고조선 | era='선사시대' OR '고조선' |
  | 삼국시대 | era='삼국시대' |
  | 통일신라·발해 | era='신라시대' |
  | 고려 전기 | era='고려시대' AND startYear < 1170 (무신정변) |
  | 고려 후기 | era='고려시대' AND startYear ≥ 1170 |
  | 조선 전기 | era='조선시대' AND startYear < 1500 |
  | 조선 중기 | era='조선시대' AND 1500 ≤ startYear < 1700 |
  | 조선 후기 | era='조선시대' AND startYear ≥ 1700 |
  | 개항기 | era='근대' AND startYear < 1910 |
  | 일제강점기 | era='근대' AND startYear ≥ 1910 |
- 세분 기준 = 한국사 학계 통설 + 한능검 출제 단위 (분량 큰 시대만 분할)
- `difficulty`: 1~5 정수
  - **1**: 가장 쉽고 기본적, 한능검 기본에 단골 (광복, 임진왜란 발발, 고려 건국, 3·1 운동 등)
  - **2**: 한능검 기본 표준 출제 수준
  - **3**: 한능검 심화 평이~중급 출제 수준
  - **4**: 한능검 심화 변별 문제 수준
  - **5**: 심화 킬러 문제 / 매우 구체적 사건 (단양 적성비, 갑인자 주조, 갑술환국 등)
- `startYear` / `endYear`: BCE는 음수 (예: -194 = 기원전 194년)
  - **같으면 점 사건** (예: 살수대첩 612-612), **다르면 기간 사건** (예: 정유재란 1597-1598, 토지조사사업 1910-1918)
  - 타임라인 정렬·채점은 항상 **startYear 기준**
- 시대별 분포: 선사 7 / 고조선 8 / 삼국 31 / 신라 22 / 고려 36 / 조선 50 / 근대 46

### 2.3 `ROUNDS` — 사건 정렬 모드 (data.js)
```js
{
  era: '개항기',
  events: [
    { text: '강화도 조약 체결', year: 1876 },
    ...
  ],
}
```
- 현재 3라운드 (개항기 / 대한제국~국권피탈 / 일제강점기), 각 5사건
- ⚠ 아직 `{ text, year }` 형식 (다른 곳은 `{ title, description, year }` 형식과 통일 필요)

### 2.4 통합 스키마 (확정)

```js
{
  id: '고려_건국',           // 한글 슬러그(공백→_). 명칭 바뀌어도 안정
  type: '사건',              // 의미 분류 (아래 12개 중 하나)
  kind: 'keyword' | 'detail', // 게임 운영 분류 (생략시 'keyword')
  title: '고려 건국',
  description: '왕건이 후삼국을 통일하고 세운 왕조의 시작',
  startYear: 918, endYear: 918,
  difficulty: 1,
  era: '고려시대',           // raw 시대명. 표시·필터는 cardEra()가 세분 매핑
  tags: ['고려', '왕건'],   // 다른 카드의 title 참조
}
```

**type 분류 (의미 라벨, 12개):**

| 키워드성 자주 (출제 단위) | 디테일성 자주 (사실 단위) |
|---|---|
| `국가` (고려, 조선, 발해) | `인물` (왕 제외 — 이순신, 정도전) |
| `왕` (광종, 세종, 흥선대원군) | `제도` (노비안검법, 균역법) |
| `전쟁` (임진왜란, 거란 침입) | `기관` (성균관, 의금부) |
| `조약` (강화도 조약, 을사늑약) | `문화` (화랑도, 골품제) |
| `운동` (동학농민운동, 3·1 운동, 무신정변) | `유물` (갑인자, 측우기, 거북선) |
| `개혁` (갑오개혁, 광무개혁) | `관직` (영의정, 도승지) |
| `사건` — keyword/detail 혼재 (위화도 회군 등) ||

- **type은 의미 라벨**일 뿐이고 게임 운영 분류는 `kind`가 결정. 같은 type이라도 kind 다를 수 있음.
- 예: 사건 type의 "위화도 회군"은 keyword, "곽재우 의병"은 detail.

**시대(era)는 카드로 만들지 않음.** 분류 라벨로만 유지. ERAS 배열의 10개 세분 시대는 cardEra() 매핑 통해 표시.

**kind 필드 의미:**
- `keyword`: 한능검 출제에서 "(가)"에 들어가는 단위 = 출제 대상. Level 1 타임라인에 등장.
- `detail`: "(가)와 관련된 사실" = 매칭 대상. Level 2 디테일 매칭에서 출제.
- 미지정 시 'keyword'로 간주 (default, backward-compat).

**타입별 시간 의미:**
| type | startYear ~ endYear |
|---|---|
| 국가 | 건국~멸망 |
| 왕 | 재위 시작~끝 (생몰 아님) |
| 인물 | 생몰 (왕 제외) |
| 전쟁/운동 | 시작~종결 |
| 조약 | 체결 시점 (점) |
| 개혁 | 시행 기간 |
| 기관 | 설치~혁파 |
| 사건 | 발생 시점 (점 또는 단기 범위) |
| 제도 | 시행~폐지 |
| 유물 | 제작/사용 시기 |
| 문화 | 융성기 또는 추정 시기 |
| 관직 | 운영 시기 |

**왕 명명 규약:** 다른 왕조의 동명 묘호 disambig 위해 "조선 태조", "고려 태조" 등 풀네임 사용. 묘호 없는 왕(예: 광해군)은 그대로.

**ID 정책:** 명칭에서 공백을 `_`로 치환한 한글 슬러그. 명칭이 바뀌어도 ID는 유지.

**태그 무결성:** 태그 값은 다른 카드의 title이어야 함. 런타임에 미해결 태그 콘솔 경고 권장. 작성 시 CANONICAL_TAGS 리스트(추후 도입) 참조.

**문제 생성 시 자동 마스킹:** 디테일 카드의 description에 부모 태그명을 명시적으로 포함시키고(예: "정유재란 당시 이순신이 명량 해협에서…"), 매칭 퀴즈 출제 시 그 tag 단어를 "(가)"로 자동 치환.

## 3. 로컬 저장 구조 (localStorage)

**키:** `history_quiz_v1`

```jsonc
{
  "version": 1,
  "totalScore": 130,

  "stats": {                          // key = 문제 title
    "고려 건국": {
      "seen": 3,                      // 종결된 출제 횟수 (정답 또는 포기)
      "correct": 2,                   // 정답 종결 횟수
      "wrong": 4,                     // ⚠ 오답 시도 누적 (한 출제에 여러번 가능)
      "gaveUp": 1,                    // 포기 종결 횟수
      "lastResult": "correct",        // 'correct' | 'gaveUp'
      "lastHadWrong": true,           // 마지막 출제에서 도중 오답 있었나
      "lastSeen": 1716480000000       // 마지막 종결 ms
    }
  },

  "history": [                        // 종결 이벤트 로그 (최대 500개, 오래된 것부터 삭제)
    { "title": "고려 건국",  "time": 1716480001234, "result": "correct", "attempts": 0 },
    { "title": "을미사변",   "time": 1716480056789, "result": "correct", "attempts": 2 },
    { "title": "강화도 조약","time": 1716480123456, "result": "gaveUp",  "attempts": 1 }
  ]
}
```

**주의:**
- `stats.wrong`은 시도 누적, `stats.seen`은 출제 종결 횟수 — 단위 다름
- `history`엔 **종결 이벤트만** 들어감. 중간 오답 시도는 종결 이벤트의 `attempts` 필드에 합산
- 사용자가 선택한 갭, 힌트 사용 여부 등은 현재 저장 안 됨 (필요시 history 항목 확장)

**계획된 확장 (Level 2 도입 시):**
```jsonc
"history": [
  // Level 1 (기존)
  { "kind": "l1", "title": "고려 건국", "time": ..., "result": "correct", "attempts": 0 },
  // Level 2 신규
  { "kind": "l2", "detailId": "명량해전", "tappedKeywordId": "이순신",
    "time": ..., "isCorrect": true, "attempts": 1, "hintsUsed": 1 }
]
```
- 기존 항목은 `kind` 누락 시 'l1'으로 간주
- Level 2 항목은 매칭 시도(맞건 틀리건)마다 한 줄 기록 — 오답 분석·약점 키워드 도출용
- `stats[title]`은 Level 1 카드만 카운트. Level 2 매칭 통계는 별도 집계 가능 (history에서 group by)

## 4. 점수 체계

### 4.1 인게임 점수 (`totalScore`)
| 이벤트 | 점수 변동 |
|---|---|
| 정답 | **+10** |
| 오답 시도 1회 | **−10** |
| 힌트 1단계 공개 (설명) | **−5** |
| 힌트 2단계 공개 (마스킹 연도) | **−5** |
| 포기 | **−20** |

- 누적식 (세션 넘어 영구 유지)
- 100 배수에 도달할 때마다 마일스톤 → 시대 재선택 화면 강제 노출 (점수·타임라인은 유지)
- 점수는 게임 화면(view-year, view-era) 우상단에만 표시. 기록 화면에는 노출 안 됨

### 4.2 출제 단위 점수 (`history[].result`) — 평균 및 레벨 산정용
종결 이벤트마다 0~100 점으로 매핑:

| 종결 결과 | 점수 |
|---|---|
| 정답 (`attempts == 0`) | **100** |
| 정답 (`attempts >= 1`) | **50** |
| 포기 | **0** |

### 4.3 최근 평균
- 최근 50건의 출제 단위 점수의 산술 평균 (정수 반올림)
- 데이터 0건이면 `—` 표시
- 기록 화면의 "최근 평균" 카드에 표시

### 4.3a 시대별 최근 평균 정답률 (시대 선택 화면)
- 시대 카드 자체가 가로 진행도 바. 좌측에서 % 만큼 반투명 초록 배경 채워짐.
- 지표 = 최근 20개 풀이 결과의 평균 (L1 + L2 통합, 해당 시대 카드만)
- 점수 환산은 §4.2와 동일 (L1) + L2 정답=100/오답=0
- 데이터 없으면 빈 바 + `—` 표시
- 사용자가 같은 시대 반복해서 풀면서 정답률이 오르면 막대가 길어짐 → 약점 → 학습 진척 자연 가시화

### 4.4 레벨 매핑 (최근 평균 → 한국사 능력시험 등급)
| 최근 평균 | 레벨 |
|---|---|
| 90~100 | 심화 1급 |
| 80~89  | 심화 2급 |
| 70~79  | 심화 3급 |
| 60~69  | 기본 1급 |
| 50~59  | 기본 2급 |
| 0~49   | 기본 3급 |

- 누적 종결 이벤트 10건 미만이면 "측정 중" (등급 미부여)
- 기록 화면의 "나의 레벨" 카드에 표시

## 5. 게임 동작 규칙 — Level 1 (키워드 연도순, 현재 구현)

> 이 섹션은 Level 1 = 현재 "연도 모드"만 다룸. Level 2 디테일 매칭은 §8 참조.

### 5.1 흐름
1. 메뉴 [연도] → 시대 선택 화면
2. 시대 1개 이상 + 풀이 모드(`일반` / `오답 위주`) 선택 → [시작]
3. 문제 카드 출제 → 사용자 스크롤·탭 → 정답/오답/포기
4. 정답 또는 포기 시 카드가 타임라인에 영구 삽입 (배경 일부가 됨)
5. 1.3초 후 자동으로 다음 문제
6. 점수가 100 배수 도달 시 자동으로 시대 재선택 화면

### 5.2 위치 판정
```
isCorrect = (gap.above.startYear < question.startYear) AND (question.startYear < gap.below.startYear)
```
- 정렬·채점 모두 **startYear 기준** (endYear는 무시 — 기간 사건도 시작 시점으로 위치 결정)
- DOM에는 `data-year` 속성에 startYear 값을 저장 (속성 이름 유지, 값만 startYear)
- 첫 카드 위 갭은 `above = null`, `lowYear = -Infinity`
- 마지막 sentinel 갭은 `below.startYear = Infinity`

### 5.3 인서트된 카드
- 정답: 사용자가 선택한 정확한 위치에 삽입 + `.snap` 클래스 (이후 갭의 일부가 됨)
- 포기: 시간순 자동 정렬 위치에 삽입 + `.snap` + `.gave-up` 스타일 (회색·이탤릭)
- 모두 `yearUsedTitles` Set에 추가되어 같은 세션에서 재출제 안 됨

### 5.4 힌트 단계 (`hintLevel`)
| 단계 | 조건 | 표시 |
|---|---|---|
| 0 | 초기 | "언제 일어났을까요?" |
| 1 | 1차 길게 누르기 (≥500ms) | description 텍스트 |
| 2 | 2차 길게 누르기 | "힌트: 91?년" (startYear의 마지막 자리 마스킹) |
| - | 3차 길게 누르기 | **포기 처리** (단계 증가 아님) |

- 기간 사건도 startYear만 마스킹 (endYear는 안 보여줌)
- 결과 표시(정답/포기): `formatYearRange(item)` — 같으면 "918년", 다르면 "1597년~1598년"

- 다음 문제 시 0으로 리셋
- 1단계와 2단계 각각 처음 공개 시 점수 −5 (반복 노출은 추가 차감 없음)
- 길게 누른 후 떼는 동작은 답안 확정으로 간주 안 함 (peek 모드)

### 5.5 풀이 모드
- **일반 풀이**: `selectedEras` 안의 미사용(`!yearUsedTitles`) 문제에서 랜덤
- **오답 위주**: 위 조건 + `QuizStore.needsReview(title)` 가 true인 문제만
  - `needsReview` = `lastResult === 'gaveUp'` OR `(lastResult === 'correct' AND lastHadWrong)`

## 7. 미래 확장 포인트

- **백엔드 동기화**: `QuizStore` 인터페이스 유지 + 내부만 fetch API로 교체
- **사용자 식별**: 현재 로컬만 사용. 실제 OAuth 도입 시 `data-provider` 기반 분기
- **카드 ID**: 현재 `title`을 키로 사용. 제목 수정 시 데이터 유실 → 고정 `id` 필드 도입 권장 (§2.4)
- **사건 모드 데이터 통일**: `ROUNDS`의 `{text, year}` → §2.4 통합 스키마로
- **난이도 기반 출제**: 사용자 레벨에 맞춰 적정 난이도 문제 우선 출제 (현재는 시대 필터만)
- **endYear 활용**: 현재 정렬·채점은 startYear만 사용. endYear 기반 채점 모드(시작/끝/포함관계 등) 추가 가능
- **인서트 카드 영속화**: 사용자의 누적 타임라인을 새로고침 후에도 유지하려면 stats에 위치 정보 추가 필요
- **응답 상세 로깅**: history 항목에 `chosenGap`, `hintLevelUsed` 등 추가하면 학습 분석 가능

## 8. 게임 동작 규칙 — Level 2 (디테일 매칭, 계획)

> 디테일 카드를 사용자의 누적 타임라인 위 키워드 카드에 매칭시키는 게임. 5지 선다 미모방 — 차별화 포인트.

### 8.1 전제 및 데이터 요구사항
- §2.4 통합 스키마 적용 후 운영
- 키워드 카드 = `kind: 'primary'` (현재 200 사건 + 추가될 엔티티 카드)
- 디테일 카드 = `kind: 'detail'` ([sample_details.js](prototype/sample_details.js) 참조)
- 디테일 카드의 `tags`에 부모 키워드 명칭이 포함되어 있어야 매칭 가능

### 8.2 흐름
1. 메뉴 [Level 2: 디테일 매칭] → 시대 선택 (단일 시대 선택, 복수 불가 — Level 1과 다름)
2. 선택한 시대의 키워드 카드들이 타임라인에 배치된 상태로 화면 표시
   - 사용자가 Level 1에서 푼 게 있으면 누적된 상태, 없으면 안내 후 Level 1로 유도(§8.4)
3. 디테일 카드가 화면 하단에 등장
   - **시기 단서 없음** (startYear 표시 안 함)
   - description에 부모 태그명이 마스킹된 형태로 노출 (예: "(가) 당시 명량 해협에서…")
4. 사용자가 타임라인의 키워드 카드 탭
5. 채점:
   - 정답: 탭한 카드의 title이 디테일의 `tags`에 포함되어 있음
   - 오답: 포함 안 됨
6. 정답이면 디테일 카드가 그 키워드 카드에 흡수 → 카드에 작은 배지 누적 ("+1 디테일")
7. 오답이면 흔들 + 점수 차감 + 시간 거리 힌트 ("이 사건은 17세기, 선택한 카드는 10세기") + 재시도 가능
8. 모든 시도(맞건 틀리건) `history`에 기록 — 약점 분석용
9. 다음 디테일 자동 출제

### 8.3 매칭 규칙
- **다중 정답 허용**: 디테일의 `tags`에 여러 키워드가 있으면 그 중 어느 카드를 탭해도 정답 (옵션 3A)
  - 예: 명량해전의 tags = ['이순신', '임진왜란', '정유재란', '조선 선조'] → 이 4개 중 사용자 타임라인에 있는 것 어느 것이든 정답
- **타임라인에 없는 정답 키워드**: 출제 풀에서 제외 (옵션 2A) — 시대별 키워드가 충분히 누적되어야 디테일 매칭 풀이 가능
- **재출제**: 일단 매번 새로 출제(같은 디테일 재출제 안 함) + 약점 분석 결과 기반으로 가중치 ↑ (옵션 4 단순)

### 8.4 빈 타임라인 처리 (옵션 3A)
선택 시대의 키워드 카드가 사용자 타임라인에 1개도 없으면:
- 안내 메시지: "이 시대 키워드 카드를 먼저 Level 1에서 학습하세요"
- [Level 1로 이동] 버튼 제공 → 해당 시대로 자동 선택된 Level 1 시작

### 8.5 힌트 단계 (Level 2 전용 — Level 1과 다름)
| 단계 | 조건 | 표시 | 점수 |
|---|---|---|---|
| 0 | 초기 | description (부모 태그 마스킹된 상태) | — |
| 1 | 1차 길게 누르기 | 세기/반세기 단서 ("16세기 말") | −5 |
| 2 | 2차 길게 누르기 | description 내 다른 키워드 1개 노출 (부분 마스킹 해제) | −5 |
| - | 3차 길게 누르기 | **포기**: 정답 키워드 자동 매칭 + 큰 점수 차감 | −20 |

### 8.6 점수 (Level 1 통합)
- §4.1 인게임 점수 체계 그대로 사용 (정답 +10 / 오답 −10 / 힌트 −5 / 포기 −20)
- `totalScore`는 Level 1·2 통합 누적 (옵션 5)
- 마일스톤(100 배수) 트리거도 통합

### 8.7 시각화 (UX 가이드)
- 키워드 카드에 매칭된 디테일 수 배지 — "디테일 3/?" 또는 작은 점 누적
- 키워드 카드 탭 시 펼쳐서 연결된 디테일 목록 보이기 (학습 모드)
- 매칭된 디테일이 누적될수록 사용자만의 한국사 사전이 만들어지는 느낌

### 8.8 알려진 한계 / 검토 필요
- **긴 타임라인 탐색 비용**: 시대별 분리로 어느 정도 완화. 시대 내 키워드가 30개 이상이면 여전히 스크롤 부담 — 화면 디자인에서 보완 필요
- **discriminative 신호 부재**: 어떤 디테일이 강한 단서인지/약한 단서인지 binary tag만 있음. 알고리즘이 자동 계산할지(tag 희소도 기반) 수동 weight 부여할지 추후 결정
- **다중 정답의 학습효과 평가**: 같은 디테일을 여러 키워드에 매칭시키도록 유도(보너스)할지, 1회 매칭으로 끝낼지 사용성 테스트 필요
- **재출제 정책 정교화**: 단순 가중치 우선 → 검증 후 SRS 등 도입 검토
